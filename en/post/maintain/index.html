Blog Maintenance Approach

2021/7/1 3-minute read

In view of my working by fits and starts，it is hard to predict the next time I update this blog, hence I record the approach to maintaining the blog, just in case.

Overview

This blog is simultaneously maintained on my Github Page and Personal Server with master and server branches on ifarewell.github.io respectively. In fact, these two pages can be merged by redirecting on to the other. But I still decide to maintain them in this slightly complex way in order to practice branch control functions in Git.

This blog is established with Hugo and its Anatole theme. And the main distinction between these two branches is introduced by the baseURL parameter in configuration file config.toml, which should be set to the actual address of the blog. And the difference of URL leads to the differences of multiple files in the generated public directory.

Post Creation

Create a new post with Hugo in the root directory of the blog project.

hugo new content/zh-cn/post/filename.md

With this instruction, Hugo will create new post with information including title, date, discription and tags according to the template file archetypes/default.md.

Page Preview

The performance can be previewed with hugo server after the post creation.

hugo server -D

With this instruction, we can preview the updated blog page at http://localhost:1313, where the newest undates will be rendered in real time.

Page Generation

With all the posts finished, we use hugo to generate pages in the directory public.

hugo --config configFilename1.toml, configFilename2.toml

This instruction creates pages with the given configuration files. And when no file is given, it automatically seek configuration file config.toml or configuration directory config/_default to conduct the generation process (according to my experience, the former is with higher priority).

It is worth noting that during local debugging, the setting of baseURL in the configuration file config.toml seems to have no effect on the results. However, it needs to be set as the address of the actual page before it can be deployed to the gitHub page or personal server for generation.

Page Deployment

After the generation, enter the public directory and synchronize the folder to GitHub using Git.

# Directory initialization before first commitment
git init

# Link the local repository to the remote origin
git remote add origin git@github.com:ifarewell/ifarewell.github.io.git

# Check updates
git status

# Add all updated files
git add . 

# Commit changes to the local Git repository
git commit -m "commit message"

# Push changes to the remote origin
git push -u origin master/server

Personal Server Deployment

With the above process, the generated pages are committed to Github, and can be access at Github Page . But for personal server, there is a one more step, that is, to pull the updates to the server.

git pull origin server:master

Here master is the name of local branch on the server, and should be distinguish with the remote origin master branch.

So far, a new post is successfully created, write and deployed to my Github Page and Personal Server . I hope this blog post is helpful to all the readers. And I also wish I could update my blog diligently and skillfully and refer to this backup blog post as little as possible🤪🤪.
